# 家庭備蓄を地域自治体から見える化するプラットフォーム

# このプロジェクトは何をするためのものか
- 各家庭の家庭備蓄状況を管理
- 自治体単位で、その自治体の各家庭の家庭備蓄状況を匿名集計
- 家庭備蓄を促進するため、プラットフォーム上で、備蓄品の消費・購入の管理
- 自治体がプラットフォーム経由の備蓄品購入をクーポンなどで支援

# なぜこのプロジェクトが必要なのか

- 国として、家庭備蓄を推奨している
- ローリングストックを推奨している
- ハードルが高く、取り組めていない人が大多数
- このプラットフォームで促進
- 地域として、各家庭備蓄の状況がわかることで、自治体自身が管理する備蓄品の最適化を行うことができる。

# 想定ユーザ
- 一般ユーザ（家庭の誰でも）
- 自治体職員
- どちらも非エンジニアを想定

## スコープと境界

| 観点 | 現状 | 決めること |
|------|------|------------|
| 「家庭備蓄」の定義 | 国の推奨・ローリングストックに言及のみ | 対象品目（食料・水・日用品・医薬品など）、数量単位、期限管理の粒度 |
| プラットフォームの範囲 | 管理・集計・購入支援 | 在庫の「記録のみ」か「EC連携まで」か、自治体備蓄との連携範囲 |
| 自治体の単位 | 「自治体単位」と記載 | 市町村か、広域連合・都道府県も含むか |

## ユーザーと権限

- **一般ユーザー「家庭の誰でも」**
  - 1家庭 = 1アカウントか、複数メンバーか
  - 招待・権限（閲覧のみ / 編集可）の有無
  - 自治体への紐付け方法（住所・住民票・自治体コードなど）
- **自治体職員**
  - アカウント発行の主体
  - 閲覧範囲（担当区域のみか、全市町村か）
  - 個別家庭データへのアクセス可否（匿名性と直結）

## 「匿名集計」の意味

README には「各家庭の備蓄状況を管理」と「自治体単位で匿名集計」が並記されている。自治体側から見える情報の粒度が未定。

1. 個別家庭は見えず、統計のみ（例：「3日分未満が40%」）
2. 個別家庭は見えるが個人名は伏せる（例：「A地区・世帯番号123」）
3. オプトインで詳細共有

プライバシー・法務（個人情報保護法）・利用者の信頼に直結するため、早めの明文化が必要。

## 備蓄品の消費・購入管理

- 手入力か、バーコード / レシート連携か
- 「購入」はプラットフォーム内ECのみか、店舗購入も記録するか
- ローリングストックのアラート（期限切れ・在庫不足）まで含むか
- 自治体支援クーポンの対象商品・対象店舗を誰が定義するか

## 自治体のクーポン支援

- 支援の形態（割引クーポン、ポイント、補助金申請の代行など）
- 予算・上限・対象世帯（全員 / 低備蓄世帯のみ）
- 自治体ごとに設定が異なるか、全国共通ルールか
- 不正利用（転売など）への対策

## 非機能・運用

- 個人情報・位置情報の取り扱い
- 可用性・SLA（災害時に使う想定なら特に重要）
- オフライン / 停電時の利用
- データの保存期間・削除ポリシー
- 障害時・災害時の自治体向けレポート形式

## 成功指標・優先順位

- 家庭の備蓄促進 vs 自治体の可視化、どちらを優先するか
- MVP で最初に出す機能（記録のみ / 集計のみ / クーポンまで）
- 「大多数が取り組めていない」理由への具体的な仮説（知識不足 / コスト / 面倒 など）

## 用語・前提の曖昧さ

- **「管理」**：家庭側の自己管理か、自治体による管理・介入も含むか
- **「最適化」**：自治体備蓄の在庫調整か、支援施策の配分か、両方か
- **非エンジニア想定**：UI の簡素化のみか、オンボーディング・ヘルプ・自治体向けダッシュボードの説明まで含むか

## 優先して決定したい3点

1. **自治体が見える情報の粒度**（完全匿名集計 vs 個別世帯の可視化）
2. **MVP の機能境界**（記録 / 集計 / 購入・クーポンのどこまで）
3. **世帯・自治体・アカウントの関係**（登録方法と権限モデル）
